# Web Internationalization Design

## Goal

The web app should support English and Chinese UI text. On first load it should choose the language from the browser when possible, allow the user to switch languages manually, and remember the manual selection for future visits.

## Scope

- Add English and Chinese translations for the React web app.
- Use `react-i18next` with i18next browser language detection and local persistence.
- Translate static UI copy, labels, buttons, table headers, filters, badges, tooltips, aria labels, empty states, and modal content.
- Keep exchange names, coin symbols, contract symbols, prices, API request parameters, trading URLs, and backend warning or error text unchanged.
- Keep backend, shared package, and API response schemas unchanged.

## Architecture

Add `i18next`, `react-i18next`, and `i18next-browser-languagedetector` to the web package.

Create an i18n initialization module in `apps/web/src/i18n.ts` that:

- Registers English and Chinese resources.
- Supports only `en` and `zh`.
- Uses local cached language first.
- Falls back to browser language detection when there is no cached choice.
- Falls back to English when the detected language is unsupported.
- Caches manual language changes in browser storage.

Create translation modules:

- `apps/web/src/locales/en.ts`
- `apps/web/src/locales/zh.ts`

`apps/web/src/main.tsx` imports the i18n initializer before rendering the app. React components use `useTranslation()` for copy and `i18n.language` for locale-sensitive formatting.

## UI Behavior

Add a language switcher to the primary navigation. It should show English and Chinese as two explicit options and keep the active language visually clear.

When the user changes language:

- Text updates immediately without a page reload.
- The selected language is persisted.
- Subsequent visits use the persisted language even if the browser language is different.

If no language is persisted:

- A browser language beginning with `zh` selects Chinese.
- A browser language beginning with `en` selects English.
- Any other language selects English.

## Formatting

Formatting helpers receive the active i18n language and use it with `Intl.DateTimeFormat` and `Intl.NumberFormat`.

USD values remain USD, exchange URLs remain English exchange URLs, and market symbols remain unchanged.

## Translated Content

Translate static frontend copy in these areas:

- Primary navigation.
- Exchange status page hero, search form, API key panel, summary cards, filters, result table, badges, empty states, auto-refresh text, and chain labels.
- Stock perpetual page hero, search form, summary cards, arbitrage lens, spread chip, result table, signal buttons, signal detail dialog, titles, and tooltips.
- Accessible labels and link titles generated by the frontend.

Do not translate:

- Exchange names.
- Coin and stock symbols.
- Contract symbols.
- Source values returned by the API.
- Warning and error messages returned by the backend.
- Local storage key names for existing credentials.

## Testing

Use test-first implementation for the web app changes.

Add or update tests to cover:

- Browser Chinese preference selects Chinese by default.
- Manual language switching updates the UI.
- Manual language selection is remembered across remounts.
- Existing coin search, filtering, API key save / clear, trading links, TradFi navigation, and signal dialog behavior still work.

Run:

- `npm run test:web -- --run`
- `npm run build`

## Risks

- `App.tsx` contains most UI and logic in one file, so translation changes can create a large diff. Keep extraction focused on text replacement and avoid unrelated component refactors.
- Tests currently assert English text. Update assertions so behavior remains covered without tying every test to one language unnecessarily.
- Backend warning text may stay English in Chinese UI because it is dynamic API data. This is intentional for this iteration.
